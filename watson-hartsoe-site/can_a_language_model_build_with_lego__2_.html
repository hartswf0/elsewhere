Can a Language Model Build With LEGO?

A picture is cheap. A build is a sequence of legal parts, exact placements, real connections, and correctable decisions. LDraw lets us tell the difference.

Watson Hartsoe Georgia Institute of Technology
Digital Media
July 2026
Atlanta, Georgia
The seduction of the finished object. A gallery of generated brick models makes the answer look obvious. The difficult question begins when we ask whether each object can be parsed, assembled, tested, and revised as a build. Image: Kulits and Schmid, BrickNet, 2026.

Put a language model in front of a box of LEGO bricks and it will sound like a builder almost immediately. It knows the names of castles and hinges, spaceships and arches. It can describe a cantilever. It can produce a charming picture of a red truck. None of this means that the truck exists as an assembly.

LEGO makes an unusually honest test for artificial intelligence because the distance between description and construction cannot be hidden for long. A brick is not a cloud of plausible pixels. It has a catalog number, dimensions, connection points, orientation, and a place in a sequence. Put it half a stud to the left and the model may not close. Rotate an axle incorrectly and the next piece has nowhere to go. Add a beautiful roof before the interior is complete and a human hand cannot finish the build.

LDraw turns these obligations into text. That is what makes it important. In an LDraw file, a model is written as commands: this part, in this color, at this coordinate, under this transformation. The format gives language models something rare—a way to produce a physical design in the medium they already speak, while leaving every placement open to inspection.

LDraw does not make construction easy. It makes claims about construction falsifiable.

The answer

Yes, under constraint Language models can generate buildable brick structures when paired with restricted vocabularies, validity checks, physical reasoning, and structured representations.
No, not by fluency alone Current general models remain unreliable at exact part selection, pose, connectivity, and long assembly sequences.

01 · The object

The hardest toy in the room.

The appeal of asking a language model to build with LEGO is easy to underestimate. The toy feels finite. Every brick comes from a catalog. Every model is discrete. Unlike clay or a neural mesh, there is no ambiguity about where one part ends and another begins.

But that neatness is exactly what makes the task severe. A convincing image can tolerate uncertainty. A brick assembly cannot. The model must choose a legal part, place it on the LEGO lattice, orient it correctly, avoid collision, connect it to the existing structure, and leave a path by which later parts can be added. The build must survive gravity. It must also survive the builder.

This is where LEGO separates symbolic competence from spatial competence. A language model can know what an arch is and still place the two bricks beneath it at incompatible heights. It can know that wheels belong on a car and still rotate an axle through the chassis. The nouns are familiar. The relations are unforgiving.

Why LEGO? The parts are standardized, the errors are inspectable, and a digital result can be checked against physical assembly.
02 · The language

A brick written as a line.

LDraw was created in the nineteen-nineties, long before large language models. Its accidental gift to contemporary AI is that a three-dimensional construction can be written as plain text without ceasing to be a construction.

The central command is Line Type 1, a reference to a part or submodel. It names a color, a position, a three-by-three transformation matrix, and a file from the parts library. A red two-by-four brick at the origin can be written like this:

1 4 0 -24 0 1 0 0 0 1 0 0 0 1 3001.dat
1Place a part
4Red
0X position
-24Y position
0Z position
1 0 0 0 1 0 0 0 1Rotation and scale
3001.datBrick 2 by 4
One line becomes one placed part.

The line is compact, but it is not vague. Change the part number and the brick changes. Change the coordinates and it moves. Change the matrix and it turns. The instruction can be rendered, compared with its neighbors, placed in a build step, and sent to another program. An MPD file can gather several LDraw models and submodels into one document.

This gives language models a tractable output language, but not a free pass. Syntactic validity is the lowest bar. A perfectly formatted line can still place a brick in midair or through another brick. LDraw can state a build precisely while stating a bad build.

Grammar answers whether the machine wrote LDraw. Construction answers whether the LDraw describes something that can stand, connect, and be assembled.
Official specification LDraw files contain one command per line. Type 1 places a referenced part using color, coordinates, and a transformation matrix.
03 · The present

What language models can build now.

The answer has changed quickly. In 2025, researchers introduced BrickGPT, an autoregressive system that treats the next brick as a next-token prediction problem. The model does not act alone. Each proposed brick passes through checks for valid type, collision, and physical stability. When a sequence becomes unstable, the system rolls back and resumes from an earlier stable state.

This is real progress. It is also a revealing concession. The model becomes a builder by giving up the fantasy that next-token fluency is enough. It needs a parts vocabulary, a coordinate system, a collision checker, a stability test, and permission to undo its own predictions.

In 2026, Brick-Composer tested multimodal language models on brick selection and exact pose. The baseline systems were poor builders: strict success at the individual assembly step was below one percent. Targeted training raised it to about fifteen percent, and one trained eight-billion- parameter model could complete as much as forty-two percent of the steps for a whole object. The leap is significant. The remaining gap is larger.

So, can language models build? The strongest answer is yes, but only inside a larger machine. The model proposes. The representation constrains. The checker rejects. The physics engine vetoes. A language model can participate in building before it can be trusted as the whole builder.

04 · The hard part

The coordinates are not the construction.

Predicting an absolute position for the next brick works tolerably in a small, regular world. It becomes brittle when the parts multiply: hinges, axles, clips, ball joints, plates, tires, decorated elements, and assemblies nested inside assemblies. One small coordinate error can be carried through every later placement.

BrickNet attacks this problem by changing what the language model predicts. Instead of asking for every part as an isolated pose in three-dimensional space, it represents a build as a graph of connections. A new part is placed relative to a part already in the assembly: this stud to that hole, this axle through that connector, this hinge at this rotation.

This is the deeper answer to the question. A pile of coordinates is not yet a construction. A construction is a history of relationships that leaves the object connected after every step. BrickNet trained on more than one hundred thousand human-designed LDraw objects and scenes because LDraw preserves the diversity that simplified voxel bricks erase.

The language model does better when it speaks less like a surveyor and more like a builder: attach this part to that part, through this connection, with this remaining freedom.

05 · The failure

The model that looked right and forgot its bricks.

A separate experiment exposed a stranger failure. An LDraw model was loaded into a renderer as one large document. The loader sorted geometry by material and merged it into efficient buffers. The assembly appeared on screen. Then the software tried to answer a simple question: which source line produced this visible stud?

It could not. Red triangles from different bricks had been combined into the same red mesh. The surface survived; the assembly history did not. Many sampled studs appeared to come from the same line, a symptom known in the project as the Line 22 collapse.

The repair was blunt: load one source line into one explicit group and attach the line number before geometry was added. Rendering became more expensive. Selection became truthful.

group.userData.lineNum = line.index;

This bug matters because it reveals a hidden temptation in AI construction. We may judge a system by the object at the end and ignore what had to be preserved to change the object later. A language model can produce valid LDraw, and a renderer can display it, while the authoring system quietly destroys the correspondence between instruction and part.

A build is not complete when it renders. It is complete when its parts can still be found, explained, and changed.

06 · The proposal

The LDraw Building Test.

The Turing test asks whether a machine can sustain the appearance of intelligence in conversation. LEGO invites a less theatrical question. Can the machine leave behind a file that another system, another researcher, and eventually another pair of hands can build?

The LDraw Building Test treats “can build” as a ladder rather than a yes-or-no performance. Each level rules out a different kind of bluff.

1. Parse The file follows LDraw grammar and references legal parts. No invented syntax
2. Place Parts have valid poses, do not collide, and respect the brick lattice. No impossible geometry
3. Connect Each added part joins the existing structure through a legal connection. No floating bricks
4. Assemble The order can be carried out by a hand or robot without blocked insertions. No impossible sequence
5. Repair A selected brick can be traced to its source line and changed without losing accepted work. No disposable world

Recent systems have begun to clear different sections of this ladder. BrickGPT demonstrates stable, manually and robotically assembled results inside a limited parts vocabulary. BrickNet expands the problem to thousands of LDraw parts and richer connection types. Brick-Composer shows how far multimodal models remain from reliable step-by-step assembly.

The final rung—repair—is where LDraw becomes more than an output format. It offers a record of the build at the same granularity as the build itself. The machine’s claim can be checked line by line. A wrong brick need not send the whole object back into the lottery of generation.

Question Evidence Failure
Did it write a model? The LDraw file parses and parts exist. Malformed command or invented part.
Did it make a structure? Parts are connected, collision-free, and stable. Floating, intersecting, or collapsing assembly.
Can it be built? A human or robot can follow the order. A later part requires passing through an earlier one.
Can it be authored? A visible brick maps back to a revisable source line. The render survives after the edit boundary is gone.

Can a language model build with LEGO? Yes—but not because it can imagine a castle. It can build when its words become legal parts, when those parts enter a connected sequence, when the sequence survives physical judgment, and when the finished object remembers enough of its making to be corrected.

The first render proves that the model can propose. The first successful repair proves that the system can support building.
The contribution A five-level test that separates formatted text, valid geometry, connected structure, feasible assembly, and repairable authorship.
